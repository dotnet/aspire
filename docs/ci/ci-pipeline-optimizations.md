# CI Pipeline Optimizations

This document tracks CI pipeline optimizations made on the `run-tests-cleanup` branch, their expected impact, and how to validate them against baseline measurements.

## Optimizations

### 1. Parallelize CLI archive build with package build

**Commit:** `3384f51628` — Parallelize CLI native archive build with build_packages

**Problem:** CLI archive build previously depended on `build_packages`, meaning it couldn't start until packages were done (~6.5 min). The CLI build then took another ~5-9 min on top.

**Change:** Made `build_cli_archives` run in parallel with `build_packages` by having the CLI build produce its own RID-specific NuGet packages locally (via `BuildBundleDepsOnly`) instead of downloading them from the package build job.

**Expected improvement:** CLI builds start immediately instead of waiting ~6.5 min for packages.

**How to validate:** In the CI timeline, `Build CLI (*)` jobs should start at roughly the same time as `Build packages`, not after it.

### 2. Use 8-core runner for Linux CLI build

**Commit:** `d1a294cb18` — build cli with 8core

**Change:** Upgraded the Linux CLI archive build runner from `ubuntu-latest` to `8-core-ubuntu-latest`.

**Expected improvement:** Faster Linux CLI build time (the most latency-sensitive build since polyglot and E2E tests depend on it).

**How to validate:** Linux CLI build should complete faster than the previous ~5 min baseline.

### 3. Remove redundant extension build from build-packages

**Commit:** `50d45910da` — Remove redundant extension build from build-packages job

**Problem:** The VS Code extension was being built in both `build_packages` and the separate `extension_tests_win` job.

**Change:** Removed the extension build from `build_packages` since `extension_tests_win` already handles it.

**Expected improvement:** Slightly faster `build_packages` completion (less work in the job).

**How to validate:** `build_packages` duration should decrease compared to baseline.

### 4. Split CLI archive build: unblock linux-only consumers

**Commit:** `90c8065cae` — Split CLI archive build so linux-only consumers aren't blocked by win/mac

**Problem:** Both polyglot validation and CLI E2E tests only need the `linux-x64` CLI archive, but they depended on the entire `build_cli_archives` job which builds all 3 OSes. The Windows build (~9 min) was the slowest, blocking these tests ~4 min after the Linux build (~5 min) finished.

**Change:**
- Parameterized `build-cli-native-archives.yml` to accept a `targets` JSON input
- Split the single `build_cli_archives` call in `tests.yml` into:
  - `build_cli_archive_linux` — linux-x64 only (polyglot + CLI E2E depend on this)
  - `build_cli_archives_other` — win-x64 + osx-arm64 (runs in parallel, nothing blocks on it)

**Expected improvement:** ~4 min reduction in time-to-first-polyglot/E2E-test. These tests should start as soon as `build_cli_archive_linux` and `build_packages` are done, rather than waiting for the Windows CLI build.

**How to validate:**
- Polyglot and CLI E2E tests should start at ~max(build_packages_end, linux_cli_build_end) instead of ~max(build_packages_end, windows_cli_build_end)
- With current timing: tests should start around the ~6.5 min mark (when packages finish) instead of ~9 min (when Windows CLI finishes)

### 5. Infra timing and dependency map in test results

**Commit:** `f4a5444f53` — Add infra timing and dependency map to test results summary

**Change:** Added infrastructure build timing (NuGet package build, CLI build durations) and test dependency mapping to the test results summary generated by `GenerateTestSummary`.

**Expected improvement:** Better observability — makes it easier to spot infrastructure bottlenecks in future runs.

**How to validate:** The test results summary should include a section showing infra build times and which tests depend on which infra jobs.

## Expected Cumulative Impact

| Metric | Before | Expected After | Improvement |
|--------|--------|----------------|-------------|
| Time to first polyglot/E2E test | ~9 min | ~6.5 min | ~2.5 min |
| Total wall-clock time | ~19 min | ~15-17 min | ~2-4 min |

The main savings come from:
1. CLI build running in parallel with packages (saves ~5-6 min of serial wait, partially overlapping)
2. Linux-only consumers not waiting for Windows CLI build (saves ~4 min of blocking)

## How to Compare Runs

Use the CI timeline tool to extract timing data from a run:

```bash
# Compare infra job timings
python3 -c "
import json, sys
data = json.load(open('ci-timeline-RUNID.json'))
for j in sorted(data['jobs'], key=lambda x: x.get('started_at','')):
    name = j['name']
    if any(k in name.lower() for k in ['build packages', 'build cli', 'setup for tests', 'polyglot', 'cli.endtoend']):
        print(f\"{name} | {j['started_at']} -> {j['completed_at']}\")
"
```

Key metrics to compare:
1. **Linux CLI build end time** — when `Build CLI (ubuntu-latest)` completes
2. **First polyglot/E2E start time** — when the first `Polyglot SDK Validation` or `Cli.EndToEnd-*` job starts
3. **Gap between (1) and (2)** — should be near-zero (just artifact download time)
4. **Total wall-clock time** — from first job start to last job completion
